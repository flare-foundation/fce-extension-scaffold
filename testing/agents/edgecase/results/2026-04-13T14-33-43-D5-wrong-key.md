# [Edge Case] D5-wrong-key
**Date:** 2026-04-13 14:33:43
**Scenario:** D5-wrong-key
**Duration:** 0m 0s
**Result:** EXPECTED_FAIL
**Error Clarity:** GOOD

## Expected Behavior
Set DEPLOYMENT_PRIVATE_KEY to a valid but unfunded key. Run pre-build.sh. Expected: "insufficient funds" error. Record error clarity.

## Actual Behavior
The pre-flight check (Step 0) caught the unfunded deployer immediately and exited with code 1.

Error message produced:
```
FATAL  fccutils/encoding.go:29  Error: deployer 0x82E7bae36c4D5882fC7772C9E3e914E2dEe36522 has insufficient funds (balance: 0 wei, minimum required: 10000000000000000 wei). Fund this account before deploying
```

Followed by:
```
[pre-build] ERROR: Pre-flight check failed — fix the issues above before deploying
```

## Error Message Quality
**GOOD** — The error message is excellent:
1. Names the exact deployer address (`0x82E7...6522`)
2. Shows current balance (`0 wei`)
3. Shows minimum required (`10000000000000000 wei` = 0.01 ETH/FLR)
4. Gives clear action: "Fund this account before deploying"
5. Fails fast at pre-flight — no gas wasted

## Observations
1. **Pre-flight balance check works perfectly.** The validation catches unfunded accounts before any deployment attempt.
2. **Minimum threshold is explicit.** Showing both current and required balance in wei is helpful for developers.
3. **Deployer address is derived correctly.** The INFO log shows the derived deployer address from the unfunded key, which helps the user verify they're using the right key.
4. **Consistent with D1/D3 pattern.** All pre-flight failures follow the same FATAL → die pattern, which is consistent and predictable.
