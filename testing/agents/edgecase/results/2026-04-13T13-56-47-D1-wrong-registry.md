# [Edge Case] D1-wrong-registry
**Date:** 2026-04-13 13:56:47
**Scenario:** D1-wrong-registry
**Duration:** 0m 17s (first attempt used real addresses due to .env override), 0m 0s (second attempt with correct override)
**Result:** EXPECTED_FAIL
**Error Clarity:** GOOD

## Expected Behavior
Set ADDRESSES_FILE to a file with wrong TeeExtensionRegistry address (a random valid address that is an EOA). Run pre-build.sh. Expected: deployment succeeds but registration fails. Record error message clarity.

## Actual Behavior
The pre-flight check (Step 0) caught the wrong registry address **before** deployment even started. The script exited with code 1.

Error message produced:
```
FATAL  fccutils/encoding.go:29  Error: TeeExtensionRegistry at 0x0000000000000000000000000000000000000001 has no deployed code. This address will be set as immutable in the contract constructor and cannot be changed. Check your deployed-addresses.json file — are you on the right network?
```

Followed by:
```
[pre-build] ERROR: Pre-flight check failed — fix the issues above before deploying
```

## Error Message Quality
**GOOD** — The error message is excellent:
1. It names the exact address that failed (`0x0000000000000000000000000000000000000001`)
2. It explains WHY it matters ("will be set as immutable in the contract constructor and cannot be changed")
3. It suggests a fix ("Check your deployed-addresses.json file — are you on the right network?")
4. It fails fast at pre-flight, before spending gas on deployment

## Observations
1. **Better than documented expectation:** The edge case doc expected "deployment succeeds but registration fails." In practice, the pre-flight check catches this BEFORE deployment, which is better behavior — no gas wasted.

2. **Finding: .env overrides environment variables.** The `pre-build.sh` script sources `.env` (line 23-27) using `set -a; source .env; set +a`, which overwrites any `ADDRESSES_FILE` env var set by the caller. This means `export ADDRESSES_FILE=/tmp/wrong.json && ./scripts/pre-build.sh` does NOT use the exported value — the `.env` file wins. This is a priority inversion bug: explicit env vars should take precedence over `.env` file defaults. The standard pattern is to check `${VAR:-}` BEFORE sourcing `.env`, or use a library that respects existing env vars.

3. **Stack trace included:** The FATAL log includes a Go stack trace, which is helpful for developers but might be noisy for end users.
