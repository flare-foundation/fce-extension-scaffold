# [Chaos] .env Command Injection via source
**Date:** 2026-04-13 14:41:00
**Scenario:** invented: env-injection-via-dotenv
**Duration:** 1m 45s
**Result:** INTERESTING
**Code Modified:** YES (created .env files with injection payloads in worktree)

## Modifications Made
Created `.env` files in the worktree root with various shell injection payloads. No source code was modified — only the `.env` file (which is gitignored).

## What Was Tried
All deployment scripts use `set -a; source "$PROJECT_DIR/.env"; set +a` to load environment variables. Tested whether a malicious `.env` file can execute arbitrary commands via shell expansion during the `source` operation.

## What Happened

**All 5 bash scripts are vulnerable to command injection via `.env`:**
- `pre-build.sh` (line 25)
- `start-services.sh` (line 40)
- `post-build.sh` (line 28)
- `test.sh` (line 24)
- `stop-services.sh` (line 31)

All use the pattern:
```bash
if [[ -f "$PROJECT_DIR/.env" ]]; then
    set -a
    source "$PROJECT_DIR/.env"
    set +a
fi
```

`source` (aka `.`) executes the file as a bash script, so any shell syntax (`$(...)`, backticks, pipes, etc.) in values is executed with the full permissions of the running user.

**The Go `godotenv` library is NOT vulnerable** — it parses `.env` files as key=value text without shell interpretation.

## Analysis

**Severity: Medium-High (in context)**

This is a well-known limitation of `source .env` in bash. Whether it's a "bug" depends on the threat model:

- **If `.env` is always user-created:** Low risk — users already have shell access, so injecting into their own `.env` gains nothing.
- **If `.env` might come from an external source** (template, CI artifact, shared config): **High risk** — arbitrary code execution under the user's permissions.
- **In a TEE/blockchain context:** A compromised or malicious `.env` could exfiltrate private keys, modify deployment parameters, or tamper with contract addresses silently.

**Recommendation:** Replace `source .env` with a safer parsing approach:
1. Use `grep -E '^[A-Z_]+=.*$' .env | while IFS='=' read -r key value; do export "$key=$value"; done` — this doesn't execute shell syntax
2. Or use a dedicated tool like `dotenv` or `env $(cat .env) command`
3. Or switch to Go-based config loading (godotenv is already safe)

**Additional finding from edge case agent (confirmed):** The `.env` sourcing also overrides explicitly-set environment variables, since `source` runs after any `export` in the calling environment. This means `ADDRESSES_FILE=/safe/path ./scripts/pre-build.sh` is silently overridden if `.env` contains `ADDRESSES_FILE=/other/path`.
