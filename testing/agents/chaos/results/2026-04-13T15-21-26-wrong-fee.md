## What Happened
**Build:** Compiles cleanly. The fee is just a `big.NewInt()` constant with no type-level or compile-time constraint. There is no mechanism to detect a wrong fee before runtime.

**Runtime (analysis based on code review, not live execution):**
- The `sendSayHello` contract function is `payable` and passes `msg.value` directly to `TEE_EXTENSION_REGISTRY.sendInstructions{value: msg.value}(...)` (InstructionSender.sol:85)
- Fee validation happens inside the registry contract (not in the extension's InstructionSender)
- With fee=1 wei, the registry's `sendInstructions` would revert

**Error path analysis:**
1. `sender.SendSayHello(opts, message)` would fail at `eth_estimateGas` stage with a revert
2. `DecodeRevertReason(err)` attempts to extract the revert string from the JSON-RPC error's data field
3. If that returns empty, `SimulateAndDecodeRevert` replays via `eth_call` — BUT it also uses the wrong fee (big.NewInt(1)), so the simulation would also revert and return the registry's revert reason
4. The error would be surfaced as: `"failed to send instruction: <rpc error> (revert reason: <registry message>)"`

**Positive finding:** The revert decoding pipeline is well-structured and would correctly surface the registry's error. The `SimulateAndDecodeRevert` fallback using the same fee means it will reproduce the exact same failure and decode the reason accurately.

## Analysis
**No compile-time protection against wrong fees.** This is expected — the fee is a runtime concern enforced by the on-chain registry contract. However:

1. **Hardcoded constant is fragile.** The fee appears as a magic number `1000000` in 4 places. If the on-chain registry ever changes its required fee, all 4 must be updated. A single constant or a runtime query (`registry.getRequiredFee()`) would be more robust.

2. **SendSayGoodbye lacks revert decoding.** Unlike `SendSayHello` which has a `DecodeRevertReason` + `SimulateAndDecodeRevert` fallback on both tx error and receipt failure paths, `SendSayGoodbye` just returns `"failed to send instruction: %s"` with the raw error — no revert reason decoding. This means a wrong-fee error on the goodbye path would produce a less helpful error message.

3. **Could not do live execution** due to missing local infrastructure. This limits the confidence of this analysis.

## Ideas for Next Time
- Stand up local infrastructure first (Hardhat + sim_dump) to enable live code-modification scenarios
- Test `SendSayGoodbye` specifically — it has weaker error handling than `SendSayHello`
- Try fee=0 (zero value) to see if the contract has a separate "no payment" check vs "insufficient payment"
- Try fee=999999 (off by one) to see how precise the registry's validation is
- Try a scenario that queries the registry for the required fee at runtime instead of hardcoding
