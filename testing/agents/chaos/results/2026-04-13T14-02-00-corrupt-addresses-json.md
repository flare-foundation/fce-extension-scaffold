**Bonus finding:** The Go bindings (`helloworld/autogen.go`) are not checked into the repo â€” they must be generated by `generate-bindings.sh` first. If a user clones fresh and runs `pre-build.sh`, the Go compilation fails with unhelpful `undefined: helloworld.HelloWorldInstructionSender` errors. The `pre-build.sh` script does call `generate-bindings.sh` in Step 1, but only after the preflight check in Step 0 â€” so the preflight itself will fail to compile if bindings don't exist.

## Analysis

**Two significant findings:**

1. **Missing field validation (MEDIUM):** `support.DefaultSupport` silently accepts a JSON file with missing required fields. When `TeeExtensionRegistry` and `TeeMachineRegistry` are absent, they default to zero-value addresses (0x000...000). The code only catches this later when it tries to connect to the chain (connection refused) or when `validate.AddressNotZero` runs â€” but `AddressNotZero` requires a live chain connection first (it's called after `DefaultSupport` connects). The JSON parsing layer should validate that required address fields are present.

2. **Error messages lack file context (MINOR):** When JSON parsing fails, the error messages (`unexpected end of JSON input`, `invalid character 'N'`) don't include the file path. A user with multiple address files won't know which file caused the error.

3. **Preflight can't compile without bindings (MINOR):** `pre-build.sh` Step 0 (preflight) runs `go run ./cmd/deploy-contract` which imports the generated bindings package. If bindings haven't been generated yet, the preflight fails with Go compilation errors, not a helpful "run generate-bindings.sh first" message. Reordering to generate bindings before preflight would fix this.
